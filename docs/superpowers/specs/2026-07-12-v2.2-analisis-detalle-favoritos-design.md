# Diseño — v2.2: análisis por proveedor (detalle, precio de catálogo, favoritos, orden)

Fecha: 2026-07-12
Estado: aprobado (brainstorming)

## Propósito

Convertir la tabla del directorio en una herramienta de **análisis y decisión**.
Surgió de una fricción real: buscando el Extech 475040, la columna "Mayoreo"
salía vacía (`—`) porque los instrumentos de nicho no publican precio de volumen
y el agente —correctamente— no lo inventa (el dato quedaba solo en las notas). El
usuario además no podía ver todo el detalle de un proveedor, marcar favoritos, ni
ordenar por precio.

Esta versión agrega: (1) **precio de catálogo** estructurado (distinto del
mayoreo), (2) **vista de detalle** por proveedor con todos los datos, (3)
**favoritos múltiples**, y (4) **orden y filtro** (por precio, por favoritos).

## Decisiones (del brainstorming)

- El "costo de envío" **no** es un campo buscable (depende de origen/destino/
  cantidad/peso): no se modela como dato del agente. En el detalle se muestra
  como *"se pide en la cotización"*. La misma lógica aplica al mayoreo real de
  productos de nicho: se consigue con **Cotizar**, no se inventa.
- **Precio de catálogo** (`catalogPrice`): precio de lista/unitario publicado. Es
  lo que el agente puede llenar para instrumentos (ej. el ~$439.99 de Master
  Supply). Distinto del `wholesalePrice` (mayoreo confirmado). Se muestra
  etiquetado como "catálogo" para no confundir con mayoreo.
- **Favoritos múltiples**: se pueden marcar varios proveedores.
- El **detalle se abre clickeando el nombre** del proveedor.
- Los 29 proveedores ya en el directorio **no** tienen los campos nuevos hasta que
  una búsqueda futura los re-encuentre. Es esperado (no se hace backfill mágico).

## Componentes

### 1. Dominio — `src/domain/supplier.ts`

`SupplierCandidate` (lo que produce el sourcing) gana:

- `catalogPrice?: number` — precio de catálogo/lista por unidad (usa el mismo
  `currency` y `priceUnit` existentes).
- `address?: string` — dirección o ciudad publicada del proveedor.

`Supplier` (persistido) gana un campo de **gestión manual** (como `status`/
`notes`, que el sourcing nunca pisa):

- `favorite?: boolean` — marcado por el usuario. Ausente/`false` = no favorito.

### 2. Sourcing — `src/sourcing/supplierSchema.ts` + `supplierSource.ts`

- El prompt del sistema pide dos campos nuevos **opcionales**: `catalogPrice`
  (precio de lista publicado, con su moneda) y `address` (dirección/ciudad si el
  proveedor la publica). Instrucción explícita: **no inventar**; omitir si no está
  publicado. Se mantiene la regla actual de mayoreo (si no hay, no penaliza).
- `parseSuppliers` valida los dos campos de forma defensiva: `catalogPrice` número
  positivo o ausente; `address` string no vacío o ausente.

### 3. Store y API — `src/directory/store.ts` + `src/server/api.ts`

- `SupplierPatch` (de `store.ts`) gana `favorite?: boolean`. `updateSupplier` lo
  aplica igual que `status`/`notes` (inmutable, refresca `lastSeen`).
- `mergeSuppliers`: al re-avistar, `favorite` se **conserva** del existente (como
  `status`/`notes`); el sourcing no lo toca. `catalogPrice`/`address` sí se
  actualizan desde el candidato cuando vienen (misma regla que el resto de datos:
  lo que el candidato trae, actualiza; lo que no trae, se conserva).
- El schema zod de lectura (`supplierSchema` en `store.ts`) suma
  `catalogPrice?`, `address?`, `favorite?` (todos opcionales → los directorios ya
  guardados cargan sin ellos).
- `patchSchema` de la API acepta `favorite` (booleano) además de `status`/`notes`;
  el PATCH sigue exigiendo al menos un campo presente.
- El CSV (`toCsv`) suma columnas `catalogPrice`, `address`, `favorite`.

### 4. Directorio público — `src/directory/publicDirectory.ts`

- `PublicSupplier` suma `catalogPrice?` y `address?` (datos útiles y públicos).
- `favorite` **NO** se publica (es una marca privada de trabajo, como las notas).

### 5. Frontend — `web/` (detalle, favorito, orden)

- **Detalle (modal):** clickear el nombre del proveedor abre un panel con todo,
  organizado en secciones: Precios (mayoreo + catálogo + MOQ + unidad), Stock,
  Dirección/región, Confianza/estado, Contacto completo, Notas, y un bloque
  "Envío: se consulta en la cotización". Acciones dentro del detalle: Cotizar ·
  Completar datos · ★ Favorito · cambiar Estado. Reusa el patrón del modal de
  cotización existente. Todo dato externo se escapa con `esc()` (XSS).
- **★ Favorito:** botón estrella por fila (y en el detalle). Toggle → `PATCH
  /api/proveedor/:key { favorite }`. Marcado visualmente (estrella llena/vacía).
- **Barra de orden y filtro** (client-side, junto al filtro de texto/estado
  actual):
  - Select **"Ordenar por"**: `precio` (menor primero; el precio efectivo es
    `wholesalePrice ?? catalogPrice`, los sin precio al final), `favoritos`
    (favoritos primero), `nombre`, `reciente` (por `lastSeen`).
  - Checkbox/toggle **"solo favoritos"**.
  - El orden/filtro se aplican sobre el arreglo del directorio en memoria; no
    tocan la API (misma arquitectura que el filtro actual).
- La columna "Mayoreo" de la tabla se mantiene; el precio de catálogo se ve en el
  **detalle** (no se agrega columna nueva a la tabla, para no recargarla).

## Flujo

```
Buscar → el agente ahora puede traer catalogPrice y address → merge/persiste.
Tabla → clic en el nombre → modal de detalle (todo el proveedor).
★ en fila o detalle → PATCH favorite → persiste en Redis/archivo.
Barra "Ordenar por" / "solo favoritos" → reordena/filtra la vista (client-side).
```

## Testing (TDD)

- **Dominio/schema (`store.test.ts`):** round-trip con `catalogPrice`/`address`/
  `favorite`; un directorio viejo sin esos campos carga igual; `mergeSuppliers`
  conserva `favorite` al re-avistar y actualiza `catalogPrice`/`address`.
- **`updateSupplier`:** aplica `favorite` (true/false), inmutable, refresca
  `lastSeen`.
- **Sourcing (`supplierSchema.test.ts`):** parsea `catalogPrice` (número) y
  `address` (string); descarta `catalogPrice` no numérico o ≤ 0 y `address`
  vacío; ausencia → campo omitido.
- **API (`api.test.ts`):** `PATCH { favorite: true }` persiste; el CSV incluye las
  columnas nuevas; `/api/publico` incluye `catalogPrice`/`address` y **no**
  `favorite`.
- **Frontend:** sin tests automatizados (vanilla); verificación manual del modal,
  la estrella y el orden en el smoke final.

## Criterios de éxito

- Clic en el nombre abre un detalle con todos los datos del proveedor.
- El precio de catálogo aparece (en el detalle) cuando el agente lo encuentra; el
  envío se muestra como "se consulta en cotización".
- Se pueden marcar varios favoritos; persisten y sobreviven a nuevas búsquedas.
- La vista se puede ordenar por precio y filtrar por favoritos.
- El directorio viejo (sin campos nuevos) sigue cargando sin romper.
- La base pública muestra catálogo/dirección; nunca los favoritos.

## Riesgos y límites

- **Datos incompletos:** muchos proveedores no publican catálogo ni dirección; los
  campos quedarán vacíos a menudo (honesto). El envío nunca se auto-completa.
- **Backfill:** los 29 actuales no ganan los campos nuevos hasta re-buscarlos; en
  Vercel la búsqueda es acotada (menos detalle). Para poblar a fondo, búsqueda
  local.
- **Deploy:** al terminar, redeploy a Vercel (push a `main`) — requiere la
  confirmación explícita del usuario, como siempre.
