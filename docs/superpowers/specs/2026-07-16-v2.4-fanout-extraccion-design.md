# Diseño — v2.4: búsqueda con fan-out de variantes + extracción estructurada

Fecha: 2026-07-16
Estado: aprobado (brainstorming)

## Propósito

Mejorar el **rinde y la consistencia** de las búsquedas sin tocar la
infraestructura. Surgió de dos observaciones en vivo:

1. **Alta variabilidad con una sola query:** la misma búsqueda del Extech 475040
   trajo 13 proveedores un día y 1 otro día; "iPhone 17" en el deploy trajo 0.
   Empíricamente, la _forma_ de la query importa muchísimo: la query exacta
   trajo 1; la variante amplia ("distribuidor mayorista de …") trajo 10.
2. **Los datos caen a `notes` en vez de a los campos:** tras el re-poblado, solo
   3/31 proveedores tenían `catalogPrice` estructurado, aunque el precio
   aparecía como texto en las notas (ej. el $439.99 de Master Supply).

Decisión del usuario: **3 variantes** por búsqueda (costo ~3x tokens aceptado).

## Componentes

### 1. Fan-out de variantes — `src/sourcing/supplierSource.ts`

- Nueva función pura **`buildQueryVariants(query: string): readonly string[]`**
  (exportada para test) con **plantillas fijas** (sin gastar tokens en
  generarlas):
  1. La query tal cual la escribió el usuario.
  2. `distribuidor mayorista de <query> en México`
  3. `proveedores de <query> al por mayor`
- `search()` corre las variantes **en paralelo** con `Promise.allSettled`
  (una variante que falla NO tumba a las demás; si TODAS fallan, se propaga un
  error como hoy). El tiempo total ≈ el de la variante más lenta → **sigue
  cabiendo en los 60s** de Vercel Hobby.
- **Dedup** de candidatos con `supplierKey` (importado de `directory/store`):
  el primero que aparece gana (mismo criterio de identidad del directorio).
- `SearchBudget` gana **`maxVariants?: number`** (default **3** también sin
  budget). Con `maxVariants: 1` se recupera el comportamiento de una sola
  llamada. El entry de Vercel y el `repoblar.ts` local usan el default (3).
- El **reintento-si-vacío** (`maxEmptyRetries`) queda **obsoleto y se elimina**
  de `SearchBudget`, del código y de los tests: el fan-out lo supera (3 tiros
  distintos > 1 repetido). `api/index.ts` deja de pasarlo.
- `enrichContact` NO cambia (una sola llamada dirigida a un sitio).

### 2. Extracción estructurada — `supplierSource.ts` + `supplierSchema.ts`

Dos capas, de menor a mayor garantía:

- **Prompt endurecido** (`buildSystemPrompt`): instrucción explícita de que
  TODO precio hallado va a `wholesalePrice` o `catalogPrice` (según sea de
  volumen o de lista) y toda dirección/ciudad a `address` — **nunca solo en
  `notes`** — con un mini-ejemplo correcto/incorrecto en el prompt.
- **Salida estructurada de la API** (mejor esfuerzo): en implementación se
  verifica —contra la documentación vigente del SDK/API— si la salida
  estructurada (JSON schema forzado) **convive con los server tools de
  `web_search`** en `claude-opus-4-8`. Si convive: se fuerza el schema de
  `{ suppliers: [...] }` y el parseo defensivo actual queda como red de
  seguridad. Si NO convive (o da fricción): nos quedamos SOLO con el prompt
  endurecido + parseo actual, sin romper nada. La decisión y su evidencia se
  documentan en el PR.

### 3. Sin cambios de infraestructura

- `/api/buscar` mantiene su contrato externo exacto (mismos campos de
  respuesta). Redis, auth, deploy, frontend y ranking no se tocan.
- El costo por búsqueda pasa a ~3x tokens (decisión explícita del usuario).

## Testing (TDD)

- **`buildQueryVariants`**: produce 3 variantes con la query interpolada; la
  primera es la query original.
- **Fan-out** (cliente mockeado): `maxVariants` default → 3 llamadas en
  paralelo; los resultados se combinan; **dedup** por `supplierKey` (mismo
  proveedor en 2 variantes aparece 1 vez); una variante que **rechaza** no
  tumba el resultado (se usan las otras); todas rechazan → error propagado;
  `maxVariants: 1` → 1 sola llamada.
- **Presupuesto**: `SearchBudget` sin `maxEmptyRetries` (tests del retry se
  eliminan); `effort`/`maxTokens`/`maxWebSearchUses` se siguen aplicando a
  CADA llamada del fan-out.
- **Prompt**: el system prompt contiene la instrucción de campos obligatorios
  (aserción sobre el string, como se testeó `localidad`).
- Si se adopta salida estructurada: test de que el request incluye el formato
  forzado; el parseo defensivo se mantiene testeado como está.

## Criterios de éxito

- Una búsqueda dispara 3 llamadas en paralelo y devuelve la unión deduplicada.
- El wall-clock de `/api/buscar` en Vercel se mantiene dentro del límite (las
  llamadas son concurrentes).
- Menos respuestas "0 nuevos" por variabilidad (3 tiros por búsqueda).
- Sube la proporción de proveedores con `catalogPrice`/`address` estructurados
  en búsquedas nuevas.
- El modo `maxVariants: 1` reproduce el comportamiento anterior (sin retry).

## Riesgos y límites

- **Costo**: ~3x tokens por búsqueda (aceptado explícitamente).
- **Rate limits**: 3 llamadas concurrentes a la API de Anthropic; para el uso
  personal actual no es problema. Si apareciera un 429, `Promise.allSettled`
  degrada con gracia (se usan las variantes que sí respondieron).
- La salida estructurada con server tools es **mejor esfuerzo**: si la API no
  la soporta en esta combinación, el prompt endurecido es el plan B (y es lo
  que se testea como mínimo garantizado).
- Productos con distribución cerrada (ej. iPhone) pueden seguir rindiendo poco:
  el fan-out mejora la cobertura, no inventa mayoristas que no publican.
