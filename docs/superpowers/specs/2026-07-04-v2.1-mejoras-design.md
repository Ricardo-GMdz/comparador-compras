# Diseño — v2.1: del buscador a herramienta de trabajo

Fecha: 2026-07-04
Estado: aprobado (brainstorming)

## Propósito

Convertir la app de sourcing (v2) de un "buscador de proveedores" en una
**herramienta de trabajo**: gestionar el directorio (estado, notas, borrar,
filtrar), redactar pedidos de cotización, mejorar la calidad del dato (unidad de
precio, enriquecer contactos), y cerrar deudas de higiene/robustez.

## A. Gestión del directorio

- `Supplier.status: "pendiente" | "contactado" | "cotizó" | "descartado"`
  (default `"pendiente"`; los directorios ya guardados sin `status` se cargan
  como `"pendiente"` — migración implícita en el schema de lectura).
- Editable desde la tabla con un select chico; **notas** editables inline;
  **borrar** con confirmación (`confirm()` del browser).
- Filtro client-side: caja de texto (nombre/material, case-insensitive) +
  select de estado ("todos" por defecto).
- Los `descartado` **no participan** del ranking ni de la mejor opción (siguen
  visibles en la tabla, filtrables).
- API nueva (el `:key` es el `supplierKey` URL-encodeado):
  - `PATCH /api/proveedor/:key` — body `{ status?, notes? }` (validado con zod).
  - `DELETE /api/proveedor/:key`.
  - Ambas devuelven envelope `{ ok, ... }`; 404 si la clave no existe.

## B. Redactar cotización

- Botón "✉️ Cotizar" por proveedor → modal con **cantidad** (número + unidad
  libre) y **especificación** (texto, prellenado con `material`).
- Genera un mensaje de pedido de cotización listo para **copiar**
  (`navigator.clipboard`), y si el proveedor tiene WhatsApp, link `wa.me` con el
  texto precargado (URL-encodeado).
- **Template local puro** (`src/quotes/quoteTemplate.ts`, sin llamada al
  modelo): saludo + presentación breve + pedido (cantidad, especificación) +
  pregunta por precio de mayoreo, MOQ y tiempos de entrega + cierre. En español.
- No se envía nada; el usuario copia y manda.

## C. Calidad del dato

- `Supplier.priceUnit: "pieza" | "kg" | "tonelada" | "m2" | "unknown"`
  (default `"unknown"`). El prompt del sourcing lo pide; el parseo lo normaliza
  defensivamente (texto no reconocido → `"unknown"`).
- **Ranking por unidad dominante**: los precios solo se comparan entre
  proveedores de la **misma `priceUnit` dominante** (mismo patrón que la moneda
  dominante del v1). Los de otra unidad o `unknown` no compiten por "mejor
  opción" por precio (pero siguen elegibles por los niveles sin precio si no hay
  alternativa). Evita el caso real "25.45/kg vs 610/pieza".
- **Enriquecer contacto** (manual, por proveedor): botón "🔍 Completar datos" →
  `POST /api/proveedor/:key/enriquecer` → el agente visita la web del proveedor
  (server tool `web_fetch`, con `web_search` de apoyo) y completa SOLO campos de
  contacto faltantes (email/phone/whatsapp/formUrl). No pisa datos existentes.
  Manual para controlar costo de tokens.

## D+E. Higiene y robustez (checklist, sin diseño)

- `BACKLOG.md` / `PROGRESO.md` del autopilot: registrar lo hecho (v1+v2) y las
  tareas reales pendientes.
- Alinear default de región UI/API (ambos `"global"`; el input de la UI queda
  con placeholder y valor inicial `global`).
- Reintento 1x del sourcing cuando devuelve 0 proveedores (nos pasó en vivo).
- Export CSV: `GET /api/directorio.csv` (servidor genera CSV con los campos
  principales; la UI agrega un link "Exportar CSV").
- Smoke E2E del server: test de integración que levanta el server real en
  puerto efímero con source mockeada y verifica servir HTML + buscar + PATCH +
  DELETE + CSV. (Nota honesta: se hace con vitest + fetch, no Playwright — para
  una app local chica no se justifica la dependencia pesada; si la app crece,
  Playwright.)
- Borrar ramas ya integradas. **Push a GitHub: requiere confirmación explícita
  del usuario en el momento** (acción hacia afuera).

## Contratos que cambian

```ts
// domain/supplier.ts
export type SupplierStatus = "pendiente" | "contactado" | "cotizó" | "descartado";
export type PriceUnit = "pieza" | "kg" | "tonelada" | "m2" | "unknown";
// Supplier gana: status: SupplierStatus (requerido en el tipo persistido,
// default "pendiente" al migrar/crear) y priceUnit?: PriceUnit.

// directory/store.ts
export function updateSupplier(
  suppliers: readonly Supplier[],
  key: string,
  patch: { status?: SupplierStatus; notes?: string },
  now: string,
): readonly Supplier[] | undefined; // undefined si la key no existe
export function removeSupplier(
  suppliers: readonly Supplier[],
  key: string,
): readonly Supplier[] | undefined;

// quotes/quoteTemplate.ts
export function buildQuoteMessage(input: {
  supplierName: string;
  material: string;
  quantity: string;
  spec: string;
}): string;

// ranking: rankSuppliers/selectBestSupplier reciben la lista ya SIN descartados
// (la API filtra) y comparan precios solo dentro de la priceUnit dominante.
```

## Criterios de éxito

- Cambiar estado/notas y borrar desde la UI persiste en `directorio.json`.
- El filtro reduce la tabla sin recargar; descartados fuera del ranking.
- "Cotizar" produce un mensaje copiable correcto (y wa.me cuando hay WhatsApp).
- Dos proveedores con unidades distintas ya no compiten por precio.
- "Completar datos" agrega contacto faltante sin pisar lo existente.
- CSV descargable; reintento de sourcing vacío; smoke E2E verde; autopilot al día.
